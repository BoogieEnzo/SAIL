# 实习汇报

## 个人说明

**个人背景与任务匹配**：简历偏软件工程、Linux 与自动化/工具链，对硬件设计与验证（RTL、形式验证等）缺乏系统训练；SAIL/ACT 任务偏 ISA 硬件语义验证，有一定错位。

**对任务的理解**：按实习方向，在 ACT 里为 Zabha 补齐测试与自动化验证，把规范变成可执行测试和可重复回归，用进度/日志留下证据链。

**AI 与协作体会**：我认为 AI agent 已经很大程度改变了开发方式；本阶段 SAIL 项目基本是在 OpenCode 驱动下完成的（在 GitHub 用 opencode 插件，issue 里发 `/oc` 即可全自动改代码、解决issue）。
我想探讨的是：agent 工具和人的领域知识如何更好协作——我认为背景知识最大的作用是**帮助清晰定义目标和验收标准**。这不代表学知识不重要：正因为我欠缺 RISC-V/工具链等背景，把很多构建与选型都交给了 AI，结果没能充分发挥工具效果，也遇到了一些问题（例如工具链方案反复、真正阻塞点发现较晚、review 时难以判断生成方案是否最优）。
因此：若有更贴合我简历背景的方向，我愿按常规流程补足知识后投入；若没有，也愿继续做 SAIL，在老师指导下补齐背景知识后再汇报一次。

**实习方向调整申请**：更感兴趣的方向是 **Linux 系统、发行版及测试**（如发行版/操作系统测试、RuyiSDK 测试、自动化测试与 CI）。若有更贴近背景且仍在招聘的 Linux/测试相关岗位，愿按常规流程补知识后转岗；或与老师们进一步沟通。

## 1. 进度结论
本阶段已完成「为 ACT 添加 Zabha 测试支持」的主体建设和前四步本地验证；**当前卡在第五步**：本机编译器不支持 Zabha 指令，无法做全量验收。有完整记录，换好工具链即可继续。

## 2. 整体开发流程
为 `riscv-arch-test` 增加 Zabha 测试支持，补齐覆盖，形成可提交 PR 的交付（代码 + 验证 + 汇报）。
任务即实习方向中的：在 **ACT（riscv-arch-test）** 里为 **Zabha** 扩展补测试并跑通自动化验证，最终交 PR 并通过 CI。本地拆成 5 步：第 1～4 步（写/生成测试、脚本分批跑、记进度）已完成；第 5 步需本机有支持 Zabha 的 GCC/binutils 做全量汇编与验收，当前环境没有，故被卡住。
